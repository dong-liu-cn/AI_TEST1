# AI 代码开发流程规范 (AI Coding Workflow)

> 原文：AIを利用してコード実装を行う時の注意...

## 1. 中文翻译对照

### 核心指令
*   **强制使用 Plan 模式**（请务必牢记以下指示）。
*   **定义结果导向的 Acceptance Criteria (AC)**。
*   **如有禁忌事项，定义 Negative Acceptance Criteria (NAC)**。
*   **建议将 Plan 导出为文件**。
*   **实现完成后，必须通过独立的会话（Session）验证是否达成 (N)AC**。
*   **向 AI 提供 Plan 文件**（理想情况下仅传递 (N)AC 部分）。
*   **利用 Playwright CLI 等 E2E 测试技能或 MCP 工具**。
*   **将验证结果导出为文件**。
*   **若验证结果为 NG（不合格），开启新会话进行修正**。
*   **提供 Plan 和评估结果**。
*   **（以此形成持续循环）**。

---

## 2. 流程详解与最佳实践

### 2.1 Plan 模式与验收标准 (AC/NAC)
*   **Plan 模式**：在与 AI 交互初期，禁止直接写代码。必须先让其输出详细的实施计划（Markdown 文件），确保思路对齐。
*   **AC (Acceptance Criteria)**：正向验收标准。即“代码做出来后，应该是什么样？”。文中强调要“仅包含结果”，意味着指令必须清晰、可量化（例如：“按钮点击后颜色变为蓝色”，而不是“优化按钮体验”）。
*   **NAC (Negative Acceptance Criteria)**：反向验收标准。用于明确“绝对不能发生什么”（例如：“页面刷新时，表单数据绝不能丢失”）。这是防止 AI 产生幻觉或越界的关键手段。

### 2.2 会话分离策略 (Session Isolation)
这是本流程中最关键的质量控制环节：
*   **问题**：如果让写代码的同一个 AI 会话去检查代码，它容易“护短”或产生盲点。
*   **解决方案**：
    1.  **Session A (Builder)**：负责根据 Plan 写代码。
    2.  **Session B (Reviewer)**：开启一个**全新的、无上下文的**会话。只给 Session B 看 **Plan + 生成的代码**。因为 Session B 不知道之前写了什么，它会像人类 Code Reviewer 一样客观地检查代码是否符合 AC/NAC。

### 2.3 自动化验证 (Playwright / MCP)
*   **Playwright CLI**：利用 AI 编写并运行端到端（E2E）测试脚本，用机器而非肉眼来验证功能是否正常。
*   **MCP (Model Context Protocol)**：利用模型上下文协议连接外部工具（如文件系统、数据库、浏览器），让 AI 能实际操作环境进行验证。

### 2.4 闭环迭代 (The Loop)
整个过程是一个严格的闭环：
`制定计划 → 实现 → 独立验证 → （如果不通过）返回修正 → 再次验证`
*   所有的 Plan 和验证结果都保存为文件，作为下一次迭代的输入，确保开发过程可追溯。
