# アプリケーション詳細設計書生成スキル

## 概要
分析済みの設計データを受け取り、ソースコードレベルのMarkdown詳細設計書を生成するスキルです。
生成される設計書は、開発者がそのまま実装に利用できるレベルの詳細度を持ちます。

## 章構成

生成する詳細設計書は以下の章構成とします：

### 第1章：基本情報
```markdown
# アプリケーション詳細設計書

## 基本情報

| 項目 | 内容 |
|------|------|
| 業務ID | {businessId} |
| 業務名称 | {businessName} |
| 機能ID | {functionId} |
| 機能名称 | {functionName} |
| 作成者 | {creator} |
| 審査者 | {reviewer} |
| 承認者 | {approver} |
| 作成日 | {createDate} |
```

### 第2章：変更履歴
```markdown
## 変更履歴

| 項番 | 変更年月日 | 変更シート | 変更内容 | 種別 | 作成 |
|------|------------|------------|----------|------|------|
| {no} | {date} | {sheet} | {content} | {type} | {author} |
```

### 第3章：ソースコード構成

| 2 | Resource | {FunctionName}Resource.java | {機能名}リソース | REST APIエンドポイント |
| 3 | 検索Logic | {FunctionName}SelectLogic.java | {機能名}検索ロジック | データ検索処理 |

### 第4章：処理詳細設計

**各処理ごとに以下の形式で記載する。これが設計書の中核部分。**

```markdown
## 処理詳細設計

### 凡例

| 記号 | 意味 |
|------|------|
| N | 正常処理 |
| E | 異常処理 |
| L | 境界処理 |
| I | I/F処理 |

### ■ {処理グループ名}（例：初期表示処理）

#### 処理{No}：{処理名}

- **クラスID**: `{ClassId}`（例：`RentalOrderInput.js`）
- **クラス名**: {ClassName}（例：レンタルオーダー入力.画面起動時）
- **メソッドID**: `{methodId}`（例：`initScreen`）
- **メソッド名**: {methodName}（例：画面初期化）
- **備考**: {remarks}（任意）

##### パラメータ

| No | パラメータ | 設定値 | 備考 |
|----|-----------|--------|------|
| 1 | {paramName} | {paramValue} | {remarks} |

##### 処理フロー

| No | 処理種別 | 処理内容 | 後続処理 | 備考 |
|----|---------|---------|---------|------|
| 1 | 処理 | {処理内容の詳細記述} | {後続処理への参照} | {備考} |
| 2 | 判定 | {判定条件} | | {備考} |
| 3 | 繰り返し | {繰り返し条件} | | {備考} |
```

**処理グループの分類（この順序で記載）：**

1. **初期表示処理** - 画面起動時のJS初期処理、Resource初期処理（新規/更新）、データ取得Logic
2. **モーダル画面の表示** - 各モーダル画面の呼び出しと戻り値処理
3. **ロストフォーカス処理** - 各入力項目のロストフォーカスイベント処理
4. **行操作処理** - 行追加、行コピー、行削除
5. **一時保存/登録処理** - 入力チェック（新規/更新）、チェックLogic、登録Resource/Logic、更新Resource/Logic
6. **帳票出力処理** - 帳票出力関連の処理（該当する場合）

## 生成ルール

### 必須ルール
1. **クラスIDは必ず記載**：各処理にクラスID（ファイル名）を明記する
2. **メソッドIDは必ず記載**：各処理にメソッドID（関数名）を明記する
3. **テーブル物理名を併記**：論理名の後に括弧で物理名を記載（例：入荷予定ヘッダ (T_RECEIVE_PLAN_H)）
4. **処理フローは表形式**：No、処理種別、処理内容、後続処理、備考の5列構成
5. **後続処理への参照**：処理間の呼び出し関係を「処理N：{処理名}」で明記
6. **変更履歴の備考反映**：変更日付と担当者を備考欄に記載

### 品質基準
- 処理フローからソースコードの骨格が読み取れること
- DTOのフィールドから画面項目とDB項目の対応が分かること
- テーブル項目マッピングから登録/更新SQLが書けること
- 共通機能参照から利用する共通コンポーネントが特定できること
