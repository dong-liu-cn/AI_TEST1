## 2. アーキテクチャ規定

### 2.1 システム全体構成

```
[ブラウザ] → [Apache HTTP Server (port 80)] → [Tomcat/WildFly (port 8080, AJP 8009)] → [Oracle DB / MySQL]
                  ↓                                   ↓
           静的コンテンツ配信              RESTful API処理
           (HTML/JS/CSS/画像)             (JAX-RS + CDI + DBFlute)
```

- Webサーバ（Apache）→ APサーバ（Tomcat/WildFly）→ DB（Oracle/MySQL）の**3層構成**
- クライアントはAngularJS/AngularベースのSPA
- サーバ通信はRESTEasy（JAX-RS）ベースのRESTful API
- CDI（Dependency Injection + Interceptor/AOP）を軸に拡張機能を提供
- Apache は `mod_proxy_ajp` で `/resources/*` パターンをAPサーバへプロキシ転送
- 静的リソース（HTML/JS/CSS/画像）はApacheが直接配信

### 2.2 クライアント側処理方式

| 規定 | 内容 |
|------|------|
| SPA構成 | 画面はAngularJSベースのSPAで制御する |
| 初期ロード | 初期ロードのみJSP（動的）、以降はHTML（静的） |
| JS/CSSロード | 初期ロード時に全てロードし、遅延ロードしない |
| 画面項目初期化 | 拡張部品を使い、サーバのDB管理定義情報で初期化する |
| サーバ通信 | 原則、AngularJSベースの非同期通信のみ |
| データ構造 | `$scope`上のデータ構造はサーバからDBのTable構造を軸に取得・利用する（クライアント側で独自管理しない） |
| 画面遷移 | メニュー画面から各業務画面への遷移はAngularJSのルーティングで制御する |
| モーダル遷移 | 別画面遷移は「モーダル」形式のみ（URLが変わらない） |
| URL直接アクセス | 各業務画面でURLを直接入力してのアクセスは原則考慮しない |
| HTTPセッション | 原則使用しない（APサーバスケールアウトに悪影響のため） |
| データ引き継ぎ | Resource → Dto(JSON) → クライアントの$scopeで管理する |

### 2.3 サーバ側処理方式

| 規定 | 内容 |
|------|------|
| API基盤 | RESTEasyベースのJAX-RSで制御する |
| DI/AOP | CDI（DI + Interceptor/AOP）を軸に拡張機能を提供する |
| クラス分類 | **Resource**（WebAPI）、**Logic**（業務ロジック）、**Dto**（データ通信用）の3種 |
| 粒度 | **1画面 = 1 Resource = 1 Dto** |
| Logic実行 | LogicはResource・Logicから実行する。クライアントから直接実行しない |
| Dto制約 | Dtoはデータのみ保持し、処理を実装しない |
| DBアクセス | 原則DBFlute経由で行う |
| トランザクション | 宣言的トランザクション管理を使用する（CDI Interceptor） |

#### 2.3.1 サーバ側クラス構成詳細

```
Resource (JAX-RS エンドポイント)
  ├── @Path("/[サブシステム]/[画面名]")
  ├── @POST / @GET メソッド
  ├── Dto を引数・戻り値として使用
  └── Logic を @Inject で呼び出す

Logic (業務ロジック)
  ├── インタフェースを必ず作成する
  ├── DBFlute BehaviorSelect/BehaviorCommand を使用
  ├── 他のLogicを@Injectで呼び出し可能
  └── Manager（共通処理）を呼び出し可能

Dto (データ転送オブジェクト)
  ├── JSON構造を定義する「容器」
  ├── フィールドのみ保持（ロジック実装禁止）
  └── 画面フィールドとの自動マッピング（双方向バインディング）
```

### 2.4 エラー処理方針（アーキテクチャレベル）

| 規定 | 内容 |
|------|------|
| システムエラー | ログ出力 ＋ ユーザへメッセージ表示 |
| 業務エラー | ログ出力せず、メッセージのみ表示 |
| TryCatch | 業務エラー以外ではTryCatch句による例外処理を実装せず、RESTEasy側に委譲する |
| もみ消し禁止 | **TryCatch句によるエラーのもみ消しやエラー内容の改竄は原則禁止** |
| 例外クラス | 業務独自の例外クラスは作成しない |

### 2.5 データ受け渡し方式

- ブラウザ↔サーバ間の通信は **JSON形式** を使用する
- データ構造はDBテーブル構造に準拠: `Resource → Dto(JSON) → クライアント$scope`
- JSON構造の標準セクション: `status`, `output`, `paging`, `data`（`head`/`body`サブセクション）
- Dto Javaファイルが「容器」としてJSON構造を定義し、Dtoのフィールドが画面フィールドに自動マッピング（双方向バインディング）

#### 2.5.1 JSON標準構造

```json
{
  "status": {
    "statusCd": 0,
    "messageList": []
  },
  "output": {
    "data": {
      "head": { /* ヘッダ情報 */ },
      "body": [ /* 明細行データ配列 */ ]
    },
    "paging": {
      "page": 1,
      "numPerPage": 100,
      "allRecordCount": 500
    }
  }
}
```

#### 2.5.2 StatusCdによる制御フロー

| StatusCd | 意味 | 処理 |
|----------|------|------|
| 0 | 正常 | 処理続行 |
| 1 | 警告（確認あり） | 確認ダイアログ表示→ユーザ承認で続行 |
| 2 | エラー | エラーメッセージ表示、処理中止 |

### 2.6 認証方式

| 方式 | 概要 |
|------|------|
| SAML認証 | IdP（Identity Provider）との連携によるSSO認証 |
| 簡易認証 | 独自のログイン画面によるユーザID/パスワード認証 |
| Google Authenticator | TOTP方式の2要素認証（ログイン後に追加認証コード入力） |

## 3. 共通規約

### 3.1 ネーミング規約（全層共通）

#### 3.1.1 機能ID
- 役割を表す名詞を連ねた名称（名詞毎のキャメルケース）
- 英字表記とする（ローマ字表記を避ける、ID連番は原則禁止）
- 30文字を超える場合は意味が通じるレベルで略す（基本的には略さない）
- 例: `SlipNoReceiveList`（伝票別入荷一覧）

#### 3.1.2 帳票ID
- 役割を表す名詞を連ねた名称（名詞毎のキャメルケース）
- 例: `InventoryWorkList`（棚卸作業リスト）

#### 3.1.3 メッセージ種別
- 4つから選択: `error`（エラー）, `warn`（警告）, `confirm`（確認）, `info`（情報通知）

#### 3.1.4 メッセージCD
- 「実態」+「現象」+「条件・理由」+「分類」（先頭小文字のキャメルケース）
- 例: `locationNotInputError`（ロケーション未入力エラー）

#### 3.1.5 区分値コード
- 全て大文字のスネークケースで単語を連ねて記述する
- 例: `DEL_FLG`（削除フラグ）

#### 3.1.6 採番コード
- 全て大文字のスネークケースで単語を連ねて記述する（基本的にテーブルの列名と一致させる）
- 例: `CONTROL_NO`（コントロールNo.）

### 3.2 DB/テーブル設計規約

#### 3.2.1 テーブル名
- **25文字以内**の英字、全て大文字のスネークケースで記述する
- 接頭辞ルール:

| 接頭辞 | 種別 |
|--------|------|
| B_ | 基盤系（ベース） |
| P_ | 基盤系（帳票） |
| M_ | マスタ系 |
| T_ | トランザクション系 |
| H_ | 履歴系 |
| W_ | ワーク系（一時表ではなく実表） |
| V_ | VIEW |

- 例: `T_RECEIVE_PLAN_H`（入荷予定ヘッダ）

#### 3.2.2 カラム名
- **25文字以内**の英字、全て大文字のスネークケースで記述する

#### 3.2.3 標準略語

| 正式名 | 略語 |
|--------|------|
| IDENTITY | ID |
| NUMBER | NO |
| CODE | CD |
| NAME | NM |
| DATE | DT |
| UPDATE | UPD |
| DELETE | DEL |
| HEAD | H |
| BODY | B |
| GROUP | GRP |
| DETAIL | DTL |

#### 3.2.4 共通カラム
全テーブルに以下の共通カラムを必須とする:
- `DEL_FLG` — 論理削除フラグ（`'0'`=有効, `'1'`=削除）
- `VERSION_NO` — バージョン番号（楽観ロック用）
- `ADD_DT`, `ADD_USER_CD`, `ADD_PGM_CD` — 登録日時・ユーザ・プログラム
- `UPD_DT`, `UPD_USER_CD`, `UPD_PGM_CD` — 更新日時・ユーザ・プログラム

#### 3.2.5 論理削除
- SELECTの条件に `DEL_FLG = '0'` を含める

#### 3.2.6 エンティティ名（論理ER図）
- 修飾語＋主要語の形で命名する（原則送り仮名は付与しない）
- 例: `入荷ステータス`

### 3.4 エラー処理規約

#### 3.4.1 基本方針
- 不必要にtry-catch句を作成しないこと（共通ルール）
- `catch (Exception e)` をメソッド毎に記述し、定型処理を書くような実装は**NG**
- 業務エラー以外のエラーはフレームワーク側（RESTEasy）に委譲する
- 業務独自の例外クラスは作成しない

#### 3.4.2 エラー制御フロー（新設計）
- JS → Resource → Logic(1つの統合メソッド) → Manager の構造を採用
- StatusCdで段階的に確認ダイアログを制御する
- Logicは1つのメソッドで複数チェックを順次実行し、エラー発生時にManagerでメッセージ登録する
- 旧設計（StatusCd毎にLogicメソッドを分割する方式）は使用しない

#### 3.4.3 入力チェックの実装
- クライアント側: Angular/AngularJS のフォームバリデーション
- サーバ側: Resource/Logic層でバリデーション実施
- 入力チェックエラーはstatusセクションのmessageListに格納して返却する

## 4. バックエンド規約（Java）

### 4.1 Javaコーディング基準

#### 4.1.1 ファイル構成（CODE01）
- 1ファイルに1つのpublicクラスを定義する
- ファイル名はpublicクラス名と一致させる
- パッケージ宣言は必ずファイル先頭に記述する

#### 4.1.2 命名規約（CODE02）
- クラス名: 先頭大文字キャメルケース（例: `ReceivePlanInput`）
- メソッド名: 先頭小文字キャメルケース（例: `searchReceivePlan`）
- 定数: 全て大文字のスネークケース（例: `MAX_RETRY_COUNT`）
- 変数名: 先頭小文字キャメルケース、役割を表す名称
- ブーリアン変数: `is` / `has` / `can` 等の接頭辞を推奨

#### 4.1.3 コメント（CODE03）
- 全publicクラス・メソッドにJavadocを記述する
- 複雑な処理ロジックにはインラインコメントを付与する
- TODO / FIXME コメントは解決後に必ず削除する

#### 4.1.4 インデント・フォーマット（CODE04）
- インデント: スペース4つ（タブ使用禁止を推奨）
- 1行の文字数: 明確な上限は設けないが、可読性を維持すること
- 開き中括弧 `{` は宣言行の最後に記述する
- 閉じ中括弧 `}` は宣言文の先頭とインデントを合わせる

#### 4.1.5 宣言（CODE05）
- メソッドの先頭で宣言せず、**使用する時に宣言する**（ただし可読性優先の場合はこの限りではない）
- 上位スコープの変数を下位スコープの変数で上書きしないこと
- 1つの宣言を1行に書く
  - ○ `int level;`  /  `int size;`
  - × `int level, size;`
- 単純な変数初期化は宣言時に行う

#### 4.1.6 ステートメント（CODE06）

| 制御文 | 規定 |
|--------|------|
| 単純文 | 1行に1つの文を記述する（`i++; j++;` は禁止） |
| 複合文 | 1文だけでも必ず中括弧 `{}` を使用する |
| return文 | 返り値に条件式がある場合を除き小括弧を使用しない |
| if-else | elseの後にスペース1つでif文を続ける |
| for文 | 実行文が1文でも中括弧でくくる。反復節は3つ以下 |
| while文 | 実行文が1文でも中括弧でくくる |
| switch文 | 全caseにbreak/return。全switchにdefault節を定義する |
| try-catch-finally | 実行文が1文でも中括弧でくくる |

#### 4.1.7 スコープ（CODE07）
- 変数のスコープは必要最小限にする
- クラス変数は極力避け、ローカル変数を優先する
- メソッドの引数とクラス変数が同名にならないよう極力避ける

#### 4.1.8 エラー処理（CODE08）
- 不必要にtry-catch句を作成しない
- `catch (Exception e)` をメソッド毎に定型処理として書くのは**NG**
- 業務エラー以外はRESTEasy側に委譲する
- 業務独自の例外クラスは作らない

#### 4.1.9 制御フロー（CODE09）
- 深いネストは避ける（早期returnパターンを活用）
- 複雑な条件式は変数に分解する

#### 4.1.10 品質（効率性）（CODE10）
- 不要なオブジェクト生成を避ける
- ループ内でのDB呼び出しは最小化する
- 文字列連結にはStringBuilderを使用する

#### 4.1.11 品質（保守性）（CODE11）
- マジックナンバーを使用しない（定数化する）
- メソッドの行数は適切な長さに保つ
- 重複コードは共通化する

#### 4.1.12 品質（信頼性）（CODE12）
- NullPointerExceptionを防ぐ防御的プログラミング
- リソース（ストリーム等）は必ずclose/try-with-resourcesで解放する

#### 4.1.13 品質（移植性）（CODE13）
- OS依存のパス区切り文字を直接使用しない
- 文字エンコーディングを明示する（UTF-8を標準とする）

- `manage.bat` の主要操作:

| 操作 | 説明 |
|------|------|
| regenerate | テーブル変更後のEntity再生成 |
| sql2entity | 外だしSQLからのEntity/パラメータBean自動生成 |
| load-data-reverse | テストデータのリバース取得 |
| replace-schema | スキーマ再構築 |

- 主要Tipsと対処法:
  - Injectでインスタンスがnullになる: `beans.xml` の確認、スコープ設定の確認
  - sql2entityでエラー: SQLファイルの構文確認、パラメータコメントの書式確認
  - regenerateに時間がかかる: 不要テーブルの除外設定
  - load-data-reverseがメモリ不足: JVMヒープサイズの増加 (`_project.bat` 内の設定)

## 8. 性能規約

### 8.1 性能目標

| 処理種別 | 性能目標 |
|---------|---------|
| サーバサイド処理 | **3秒以内** |
| 画面描画 | **3秒以内** |
| Excel出力（最大件数） | **30秒以内** |
| 帳票出力（最大件数） | **30秒以内** |

- サーバサイド処理とクライアントサイド処理（画面描画）の性能は**切り離して考える**
- サーバサイド: 検索処理を行い検索結果を返すまでの時間
- 画面描画: 検索結果データを受信しウィジェットにデータ設定が完了するまでの時間

### 8.2 SQL開発時の注意事項

| # | 規約 |
|---|------|
| 1 | 検索結果データは**3000件以下**になるようにする（Excel/帳票出力は対象外） |
| 2 | `SELECT *` は使用しない。必要な項目だけを記述する |
| 3 | `DISTINCT` は使用しない。使わなくて済むように設計する |
| 4 | ソート不要であれば `ORDER BY` を外す |
| 5 | 複合インデックスを使う場合、WHERE句にインデックスと同じ順番で条件指定する |
| 6 | 結合条件はインデックスが有効になるよう記述する |
| 7 | カーソルは使用しない（不可避の場合は処理が冗長にならないよう注意） |
| 8 | `LIKE '%xxx%'` のような前方一致でない曖昧検索は使用しない（使用時は性能検証必須） |

### 8.3 ER図設計のポイント

- 非正規化テーブルや冗長カラムの利用をパフォーマンス観点で検討する
- インデックス設計はクエリパターンに基づいて行う
- 検索頻度の高いカラムにはインデックスを付与する
- 複合インデックスのカラム順序はクエリのWHERE句に合わせる

### 8.4 画面描画のポイント

- `ng-repeat` は1000行を超えないようにする
- `$watch` の不要な作成を避ける
- `ng-if` と `ng-show`/`ng-hide` の使い分けを検討する（DOMを削除する方がパフォーマンスが良い場合は `ng-if`）
- 大量データの表示にはページングを活用する
- 画像やアイコンの遅延ロードは行わない（初期ロード方針に従う）

## 10. EDI・汎用データ入出力

### 10.1 EDI連携概要

#### 10.1.1 EDI（Electronic Data Interchange）機能
- 外部システムとの電子データ交換を行う機能
- 受信（インバウンド）・送信（アウトバウンド）・データ変換の3機能を提供

#### 10.1.2 EDI処理フロー
```
[外部システム] → [EDI受信] → [データ変換] → [WMS取込]
[WMS出力] → [データ変換] → [EDI送信] → [外部システム]
```

#### 10.1.3 EDIデータフォーマット
- 固定長フォーマット
- CSVフォーマット
- XMLフォーマット
- フォーマット定義はマスタで管理する

### 10.2 汎用データ入出力

#### 10.2.1 概要
- マスタデータ・トランザクションデータの一括取込/出力機能
- Excel / CSV 形式に対応
- 取込テンプレートを使用したデータ入力支援

#### 10.2.2 取込処理
- ファイルアップロード → バリデーション → データ登録 の3ステップ
- バリデーションエラーは行番号・カラム名とともにエラーリストを返却
- 大量データの取込はバッチ処理として実行可能

#### 10.2.3 出力処理
- 検索条件を指定してデータ抽出
- Excel / CSV 形式で出力
- 大量データはストリーミング出力を使用
