# ワークフローのベストプラクティス

> 効果的で信頼性の高い Cline ワークフローを作成するためのヒントと戦略。

効果的なワークフローを作成するには、明確な指示・モジュール設計・インテリジェントなツール使用のバランスが必要です。これらのベストプラクティスに従って、Cline の自動化機能を最大限に活用しましょう。

---

## Cline を使ってワークフローを構築する

ワークフローの構築に Cline を活用することを強くお勧めします。Cline はプロジェクトのコンテキストと構造を理解しているため、あなたの特定のニーズに合った自動化を設計する上で非常に価値あるパートナーとなります。

### 自分のワークフローを構築する

ワークフローの作成は、思っているよりも簡単です。実は「ワークフローを作るためのワークフロー」が存在します！

まず、[create-new-workflow.md](https://github.com/cline/prompts/blob/main/workflows/create-new-workflow.md) ファイルをワークスペースに保存します（例：`.clinerules/workflows/` の中）。

次に、`/create-new-workflow.md` と入力すると、Cline が以下のステップでガイドしてくれます：

1. 目的と簡潔な名前を尋ねます。
2. 目標と期待するアウトプットを説明します。
3. 主要なステップを列挙します（Cline が詳細を決定するのを手伝います）。
4. 適切な構造のワークフローファイルを生成します。

> 💡 **ヒント - 履歴を自動化する：**
> 最良のワークフローは、すでに完了したタスクから生まれます。繰り返す必要があることを完了した後、Cline に「今完了したプロセスのワークフローを作成して」と伝えましょう。Cline は会話を分析し、ステップを特定して、ワークフローファイルを生成します。蓄積されたコンテキストが再利用可能な自動化になります。

プロジェクト固有のワークフローは `.clinerules/workflows/` に、複数のプロジェクトで使うグローバルなワークフローは `~/Documents/Cline/Workflows/` に保存されます。名前が一致する場合、プロジェクトワークフローが優先されます。

## ワークフロー設計

> 💡 **ヒント - シンプルから始める：**
> 小さな単一タスクのワークフローから始めましょう。慣れてきたら、それらを組み合わせたり、より複雑なシーケンスを作成したりできます。

### モジュール化

一つの巨大なワークフローファイルを作るのではなく、複雑なタスクをより小さく再利用可能なワークフローに分解しましょう。これにより、保守やデバッグが容易になります。

### 明確なコメントの使用

コードと同様に、ワークフローのステップにコメントを付けることは非常に重要です。何が起きているかだけでなく、*なぜ* そのステップが行われるのかを説明しましょう。これにより、将来の保守担当者（あなた自身を含む）と Cline が意図を理解できます。

### バージョン管理

ワークフローをコードベースの一部として扱いましょう。Git リポジトリ（`.clinerules/workflows/` 内）に保存することで、バージョン管理され、レビューされ、チームと共有できます。

## Cline のプロンプトエンジニアリング

### ツール使用を明確にする

単に「ファイルを見つけて」と言うのではなく、Cline がどのツールを使うべきかを明示しましょう。

- **悪い例：** 「ユーザーコントローラーを見つけて。」
- **良い例：** 「`search_files` を使って `src/controllers` ディレクトリ内の `UserController` を探して。」

## 高度なテクニック

### 利用可能なツール

Cline にはワークフロー内で使用できる強力なツールセットがあります。最もよく使われるツールを紹介します：

#### execute_command

システム上で CLI コマンドを実行します。テスト実行・ビルド・git コマンド、その他のターミナル操作に使用します。

```xml
<execute_command>
  <command>npm run test</command>
  <requires_approval>false</requires_approval>
</execute_command>
```

#### read_file

ファイルの内容を読み込みます。コードや設定の分析に必須です。

```xml
<read_file>
  <path>src/config.json</path>
</read_file>
```

#### write_to_file

ファイルを作成または上書きします。ボイラープレートコード・設定ファイル・ドキュメントの生成に使用します。

```xml
<write_to_file>
  <path>src/components/Button.tsx</path>
  <content>
    // ファイルの内容をここに記述...
  </content>
</write_to_file>
```

#### search_files

ディレクトリ内のファイルに対して正規表現パターンを検索します。TODO・使用例・特定のコードパターンを見つけるのに最適です。

```xml
<search_files>
  <path>src</path>
  <regex>TODO</regex>
  <file_pattern>*.ts</file_pattern>
</search_files>
```

#### ask_followup_question

ユーザーに入力や確認を求めます。ワークフローをインタラクティブにし、人間による意思決定を可能にします。

```xml
<ask_followup_question>
  <question>本番環境にデプロイしますか？</question>
  <options>["はい", "いいえ"]</options>
</ask_followup_question>
```

#### browser_action

組み込みブラウザを制御して、ウェブサイトやローカルサーバーと対話します。Web UI のテストやデータのスクレイピングに便利です。

```xml
<browser_action>
  <action>launch</action>
  <url>https://localhost:3000</url>
</browser_action>
```

### MCP ツールの活用

ワークフロー内でモデルコンテキストプロトコル（MCP）ツールを使用して、GitHub・Slack・データベースなどの外部サービスと連携できます。これにより、強力なエンドツーエンドの自動化を実現できます。

### コンテキストウィンドウの管理

Cline のコンテキストウィンドウに注意してください。ワークフローが長すぎたり、処理するデータが多すぎたりすると、トークン制限を超えてしまう可能性があります。

- **分割する：** 長いワークフローを小さな部分に分割しましょう。
- **簡潔にする：** 指示を明確で要点を押さえたものにしましょう。
